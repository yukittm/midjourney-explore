The style `home base` register (the dominant "home" look) has a load-bearing decision history that recurs in design reviews of `docs/style/style-definition.md`:

- It was **deliberately deferred** (style-definition.md "Home base — deferred" section + Status block) with the rationale: it should *emerge from the user's selections over time*, and those selections double as the `--p` training signal.
- As of 2026-06-24 a formalization edit proposed flipping it to **decided = "Flowing Color Waves"** based on ONE keeper-curation event (2026-06-23). The user endorsed it as "mostly OK."
- Review judgment: the direction is right but a hard *lock* is premature — one curation event is not the "selection history over time" the deferral required, and the project has a LESSONS scar (`.claude/rules/LESSONS.md` top entry) from over-fitting analysis to too few reference images. Recommended label: **"provisional home base (center of gravity, not a cage)"**, not "decided."

**Why:** Two competing pressures — the user wants a concrete center of gravity for automation; the SSoT's own deferral clause + LESSONS scar warn against early lock-in on thin evidence.

**How to apply:** When reviewing any edit that changes home-base status, check whether the evidence is a single selection event vs accumulated selection history, and whether the `--p` training loop has actually run. Keep the "deferred"/"provisional"/"decided" wording consistent across BOTH the Home-base section AND the Status block (they have contradicted each other in draft edits).
