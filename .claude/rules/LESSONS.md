# Lessons Learned

- **Verify a premise across the FULL evidence set before building on it — and stop proactively if it's shaky**: I characterized the reference artist's "signature" and built a whole richness model (P1/P2/P3) + a strategy + multi-agent dispatch from only **2-3 cherry-picked reference images**, when the real basis is the curated **11-image** set — and the full set FALSIFIED the 2-image reading (giz is mostly **heroless / minimal**, not dominant-hero; "add density / a dominant hero" was the wrong fix). → When analysis rests on a body of evidence (reference images, files, data), examine the **whole relevant set first**; if the premise looks under-sampled or off, **STOP and verify it before spending tokens** on downstream strategy. Don't merely execute the user's instruction — **lead**: check assumptions, flag bad premises early, drive the project. Seeding sub-agents from a shaky premise multiplies the error.
- **Verify load-bearing MJ params in-app before baking them into prompt recipes**: `--sv 7` was carried from a research-doc claim **explicitly flagged "verify in-app"** straight into every prompt recipe/template, and caused a live submit error — *"Unsupported Style Reference version 7 for --version 8.1."* → A param flagged uncertain must be **tested in MJ before it enters a recipe**. (Resolved + web-re-verified: for an image-URL `--sref`, **omit `--sv`** → it defaults to **sv7**, the V8.1 default and `--hd`-compatible; numeric CODES need sv4/sv6. NOTE: my *first* fix's rationale — "sv7 rejected / sv6 breaks `--hd`" — was itself WRONG until web-verified. → verify against **authoritative sources**, don't just react to one error message; a hasty "fix" can be as wrong as the original.)
- **Archive legacy into `/archive/`, keep the live tree clean**: LLM-assisted work accumulates files; I left superseded docs in place (V7 research bundle in `docs/research/`, V7 automation idea in `automation/`) with only "ARCHIVE" banners instead of moving them, and never created an archive folder the user had asked for. → When a doc is superseded, **move it to top-level `/archive/`** (cautious deletion = *move, not delete*; preserve provenance), update `/archive/README.md` + every pointer, and return the repo to a clean current-only state. Default to pruning/relocating legacy, not letting it pile up.
- **Legacy-carryover bias on creative subjects**: When asked for new MJ prompts I repeatedly recycled prior subjects (antelope-rider, heroless cracked-earth landscapes) instead of inventing fresh ones — a preservation bias that smuggles stale intent forward. → When the user asks for new subject matter, generate genuinely NEW subjects; never reuse a specific subject/scene from earlier prompts unless explicitly told to iterate. The giz surreal-natural WORLD is the constant; the specific subjects must be fresh each time.
- **Don't narrate an agent dispatch you didn't actually emit**: twice I wrote "I launched N agents" in prose while the turn contained **no `Agent` tool call**, so nothing ran — only caught when the user asked "what's the status?". → A claim that work is "running/launched/起動した" MUST correspond to an actual tool-use block in the SAME turn. Before writing it, confirm the `Agent` call is really in the message; describing a dispatch is not performing it. Report honestly and launch for real.
- **MJ prompt length: don't ship sparse, but length is NOT the lever**: I repeatedly shipped lazy/sparse prompts; then over-corrected into a dogmatic ~40-word target. In-app (2026-06-23) a ~85-word body AND the simple ~20-word originals both produced the look. → Don't default to brevity (specify the subject + the real background noun concretely), but **don't chase a word count** — the actual levers are the realism **kernel** (`--style raw` + low `--s` + photo cues + `--no painting…`) + **naming the real subject/phenomenon** (incl. the bridge) + the **`--sref`**. See `docs/style/prompting.md` §7. (Supersedes the earlier "~40-word target, count before sending" rule.) **Update (2026-06-23): DETAIL is the lever for DIFFERENTIATION.** Length isn't needed to *hit the look* (kernel/sref/profile do that), but **rich ~80-word scene specs are the PRODUCTION DEFAULT** — under-specifying lets a mass of outputs collapse to MJ's *average* (sameness, the user's explicit fear at scale). I keep wrongly resetting to short prompts (esp. for Remix, rationalizing "the source carries it") → STOP: rich-by-default for production, short only for a style-check; use **low `--s`** so detail is honored (high `--s` re-averages). See `docs/style/prompting.md` §7.
- **Don't state unverifiable external facts as if certain (esp. metrics the user can't easily catch)**: I asserted a search-volume comparison (`"AI art" > "AI visual"`) confidently to justify a bio wording, with **no data source** — pure intuition dressed as fact. The user pushed back ("don't glibly say things you can't actually verify"). IG doesn't even publish search volumes, so the claim was unfalsifiable on my end. → For any factual claim about **external reality I have not verified** — volumes, rankings, "X performs better than Y," API/quota numbers, market data — either (a) cite a real source, (b) prefix `⚠️ 推測:` and state the basis + its limits, or (c) say "I don't know / not verifiable." NEVER let intuition pass as established fact, *especially* where the user lacks the means to check it. When a claim is load-bearing for a decision, offer to verify (web/data) rather than asserting. (Reinforces CLAUDE.md §Evidence-Based Responses.)
