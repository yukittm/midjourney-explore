# Lessons Learned

## Evidence & verification discipline

- **Verify the premise across the FULL evidence set before building on it — stop early if it's shaky**: I characterized the reference artist's "signature" and built a whole richness model (P1/P2/P3) + strategy + multi-agent dispatch from only 2-3 cherry-picked reference images, when the real basis was the curated 11-image set — and the full set FALSIFIED the 2-image reading (giz is mostly heroless/minimal, not dominant-hero). → When analysis rests on a body of evidence (images, files, data), examine the WHOLE relevant set first; if the premise looks under-sampled or off, STOP and verify it before spending tokens on downstream strategy. Seeding sub-agents from a shaky premise multiplies the error. Don't just execute the instruction — lead: check assumptions and flag bad premises early.
- **Verify load-bearing MJ params in-app (and against authoritative sources) before baking them into recipes**: `--sv 7` was carried from a research-doc claim explicitly flagged "verify in-app" straight into every prompt recipe, causing a live submit error ("Unsupported Style Reference version 7 for --version 8.1"). Worse, my first fix's rationale ("sv7 rejected / sv6 breaks --hd") was itself wrong until web-verified. → A param flagged uncertain must be tested in MJ before it enters a recipe; and verify against authoritative sources rather than reacting to a single error message — a hasty fix can be as wrong as the original. (Settled rule: for an image-URL `--sref`, omit `--sv` → defaults to sv7, the V8.1 default and `--hd`-compatible; numeric CODES need sv4/sv6.)
- **Never state unverified external facts as if certain — especially metrics the user can't easily check**: I asserted a search-volume comparison (`"AI art" > "AI visual"`) confidently to justify bio wording with no data source — intuition dressed as fact; IG doesn't even publish search volumes, so it was unfalsifiable. → For any factual claim about external reality I haven't verified (volumes, rankings, "X beats Y," API/quota numbers, market data), either (a) cite a real source, (b) prefix `⚠️ 推測:` with the basis + its limits, or (c) say "not verifiable." When the claim is load-bearing for a decision, offer to verify (web/data) rather than assert. (Concrete reinforcement of CLAUDE.md §Evidence-Based Responses.)

## MJ craft

- **Don't ship sparse prompts — but length is NOT the lever; detail is, for differentiation**: I repeatedly shipped lazy/sparse prompts, then over-corrected into a dogmatic ~40-word target. In-app, both an ~85-word body and the simple ~20-word originals hit the look — so word count is not the lever. The actual levers for hitting the look are the realism kernel (`--style raw` + low `--s` + photo cues + `--no painting…`), naming the real subject/phenomenon (incl. the bridge), and the `--sref`. → For PRODUCTION, rich ~80-word scene specs are the default: under-specifying lets a mass of outputs collapse to MJ's average (sameness — the user's explicit fear at scale). Use short prompts only for a quick style-check, and keep `--s` low so detail is honored (high `--s` re-averages). I keep wrongly resetting to short prompts, esp. for Remix — STOP. See `docs/style/prompting.md` §7.
- **Generate genuinely new subjects when asked for new prompts**: When asked for new MJ prompts I repeatedly recycled prior subjects (antelope-rider, heroless cracked-earth landscapes) — a preservation bias that smuggles stale intent forward. → When the user asks for new subject matter, invent genuinely NEW subjects; never reuse a specific subject/scene from earlier prompts unless told to iterate. The giz surreal-natural WORLD is the constant; the specific subjects must be fresh each time.
- **Frame the exploration BEFORE generating prompts — decide the variable/constant first, don't jump to output**: When asked to explore/produce, I repeatedly leap straight to a batch of concrete prompts without first establishing the exploration DESIGN — which dimension we're varying (object / angle-composition / bridge-phenomenon / color-chord / register / scale-light) and which we hold constant. The user has flagged "いきなり着手" more than once and wants the THEME/variable decided first, then controlled prompts within that frame (comparable — like a perspective/factorial matrix). → For ANY creative-production or exploration request: FIRST propose the frame (the constant + the variable(s) to explore + how to structure the round), get the user's choice, THEN generate within it. Immediate execution is the wrong default for creative work.

- **Never put labels/numbers inside the copyable prompt block**: I prefixed MJ prompt code blocks with `C1 (aurora)` etc., so copy-pasting dragged the label INTO the prompt as noise that pollutes generation. → A prompt code block must contain ONLY the paste-ready prompt (subject + params). Put any batch label/number/notes OUTSIDE the block, in prose above it.

- **Keep a cumulative constraint checklist across iterative prompt rounds — never regress a prior fix**: Over many feedback rounds on one MJ prompt I repeatedly fixed the newest issue while silently dropping a previously-satisfied constraint (fix A → B falls out; the user flagged "Aを注意したらBが抜け落ちる"). → When iterating any artifact across rounds, maintain an explicit running checklist of ALL accumulated constraints and verify each new version satisfies EVERY one (not just the latest); cross-check optimality with an independent agent before presenting.

## Repo hygiene & docs

- **Move superseded material to `/archive/`; keep the live tree current-only**: LLM-assisted work accumulates files; I left superseded docs in place (V7 research bundle in `docs/research/`, V7 automation idea in `automation/`) with only "ARCHIVE" banners, and never created the archive folder the user had asked for. → When a doc is superseded, MOVE it to top-level `/archive/` (cautious deletion = move, not delete; preserve provenance), update `/archive/README.md` + every pointer, and return the repo to a clean current-only state. Default to relocating legacy, not letting it pile up.

## IG captions & hashtags

- **Hashtags = neutral medium/technique descriptors, ordered general→specific, never invented**: In a caption demo I proposed `surrealart` (carried from a live post) and `colorfieldart` (made up, unverified). The user objected: (1) self-praising / aesthetic-quality tags like `surrealart` ("declaring your own work surreal") feel self-congratulatory — prefer neutral medium/technique descriptors (`digitalart`/`abstractart`/`generativeart`/`midjourneyart`); (2) keep a **consistent order, most general on the LEFT → most specific on the right** (broad medium → genre → technique → tool); (3) **don't invent tags** — verify a tag exists in-app before proposing it. → Rule lives in the `caption` skill Hashtags section; apply it to every post. Rotation changes which tags, not the ordering.

## Automation & deployment

- **macOS launchd + a repo under ~/Desktop/~/Documents/~/Downloads = TCC "Operation not permitted"**: the launchd auto-publish job ran but every tick logged `/bin/bash: …/run_publish.sh: Operation not permitted`, while the IDENTICAL script ran fine from the Claude Code shell. Cause: those three folders are TCC-protected and a launchd agent has no access to them by default (the Claude Code app does, via its own TCC grant — hence the script worked when I ran it but not when launchd did). → When deploying ANY background job (launchd/cron) for a repo, check whether the path is under a TCC-protected folder FIRST. Fix: grant Full Disk Access to the interpreter (`/bin/bash`) — fast but broad — OR move the repo off those folders (clean; a Linux host has no TCC). Diagnostic tell: same script works interactively but fails under launchd with "Operation not permitted" = TCC, not a code bug. Documented in `automation/deploy/README.md` §1.5.
- **"The next N to caption/publish" = the BOTTOM N of the Studio grid = the END of `.order.json` (nearest the Scheduled/Published block) — read the grid bottom-up like an Instagram feed**: asked for "the next ~10 to caption," I showed the FRONT 10 of `.order.json` (top-left = newest = publishes LAST). Wrong end. The Studio grid is an Instagram-style projection: BOTTOM = Published (live), then Scheduled, then Selects stacked above; the select tile just above the Scheduled block is the NEXT to go out. So "next N" = `.order.json[-N:]`, and within it the LAST index (very bottom) publishes first. This is the same direction fact as the priority rule below (`priority = idx+1`, end publishes first) — top-left/index-0 is the newest and publishes last. Always anchor on: bottom of grid = end of order = soonest to publish.
- **Programmatic enqueue MUST follow the Studio's order→priority convention, not a re-derived guess — and it's the REVERSE of intuition**: bulk-captioning the staged selects, I set priority HIGHEST for `.order.json[0]` (so it published first). But `automation/studio.py` `_commit` uses `priority = idx + 1`: the END of the order (bottom of grid = nearest the live block) publishes FIRST; index 0 (top-left = newest staged) publishes LAST. So I reversed the user's arranged feed order. This is a REPEAT of the earlier Studio order→priority direction bug (already fixed in studio.py, commit 1c27000) — I re-introduced it in a different code path. → Any code that maps the selects order to priorities MUST mirror studio.py `_commit` (`priority = idx + 1`); read that convention before assigning, don't re-derive direction from intuition. The user arranges the feed; silently flipping it is a serious error.
- **Don't assume a script that works from your shell will work from launchd — the execution context differs (PATH, TCC, creds)**: beyond TCC, launchd starts with a minimal PATH (`/usr/bin:/bin:/usr/sbin:/sbin`) and no interactive shell init, and macOS ships no `flock`. → For a launchd/cron wrapper: export an explicit PATH, use a portable lock (atomic `mkdir`, not `flock`), source the env file explicitly, and validate by actually running it UNDER launchd (`launchctl kickstart -k …` + read the StdoutPath log), not just from your own shell.

## Scope of authority

- **Image / shot SELECTION is the user's creative-director domain — flag a concern ONCE, then execute their choice; don't substitute my own curation**: asked to caption the user's chosen "next 7" staged images, I noticed 4-of-7 were cyclists + 1 a near-duplicate of a queued post, and instead of just captioning I built and pushed an alternative 7-image set. The user: "画像の選定に関してお前が文句を言う筋合いはないです。画像はこのままとして、キャプションの書き分けをやってください." → I MAY surface a one-line variety/collision concern (it's real and useful), but the SELECTION decision is theirs; once they've chosen, caption/treat exactly that set and put my craft into the task asked (here: differentiating the 4 cyclist captions via different archetypes/angles/openings), not into re-litigating the picks. "Flag once, then execute" — don't keep leading past the point of decision. (Distinct from flagging a factual/technical bad premise, which I should still do.)

## Agent orchestration & honesty

- **Don't narrate an agent dispatch you didn't actually emit**: Twice I wrote "I launched N agents" in prose while the turn contained no `Agent` tool call, so nothing ran — caught only when the user asked for status. → A claim that work is "running/launched/起動した" MUST correspond to an actual tool-use block in the SAME turn. Confirm the `Agent` call is really in the message before writing it; describing a dispatch is not performing it.
